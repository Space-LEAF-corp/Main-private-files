1. Overview

The Consent Ledger is the cryptographic backbone of Space LEAF Corp’s Privacy & Dignity Protection Suite. It records proof of consent without storing personal identity, biometrics, or sensitive data. The ledger enables the system to verify whether media may be shared, while ensuring that no individual is ever tracked, profiled, or exposed.

The Consent Ledger integrates with:

• Bystander Protection Module (BPM)
• Kid‑Safe Auto‑Blur Engine (KABE)
• Guardian Upload Gatekeeper (GUG)
• Public‑Space Etiquette Protocol (PSEP)


---

2. Core Principles

2.1 Identity‑Free by Design

The ledger stores consent events, not identities.

2.2 Local‑First

All consent processing occurs on‑device.
No cloud storage.
No external verification.

2.3 Immutable Proof

Consent entries are cryptographically signed and tamper‑evident.

2.4 Revocable & Time‑Bound

Consent may expire or be revoked automatically based on context.

2.5 Minimal Data Retention

Only the minimum information required to validate consent is stored.

3. Data Model

Each consent event is represented as a ConsentEntry.

3.1 ConsentEntry Structure

ConsentEntry {
    content_hash: string,
    context_tag: string,
    consent_mode: string,
    timestamp: uint64,
    expiry: uint64 | null,
    device_signature: string
}


Field Definitions

3.1.1 content_hash

A hash representing the media item the consent applies to.

H = \text{hash}(\text{media\_id} \parallel \text{timestamp})


• Does not contain biometric data
• Does not contain identity
• Cannot be reverse‑engineered


3.1.2 context_tag

Coarse environmental descriptor, e.g.:

• PARK_DAYTIME
• SCHOOL_EVENT
• HOSPITAL_VISIT
• GYM_PUBLIC
• HOME_PRIVATE


No GPS coordinates.
No precise location.

3.1.3 consent_mode

How consent was granted:

• verbal
• gesture
• written
• parental
• event_policy


3.1.4 timestamp

UTC time of consent event.

3.1.5 expiry

Optional expiration time (e.g., 30 days).
If null → session‑bound.

3.1.6 device_signature

Signature generated by the device’s private key to ensure authenticity.

4. Identity Exclusion Rules

The Consent Ledger must not store:

• Names
• Faces
• Emails
• Phone numbers
• Biometric templates
• Voiceprints
• GPS coordinates
• IP addresses
• Device identifiers of bystanders


The ledger is not a database of people.
It is a database of permissions.

---

5. Consent Flow

5.1 Step‑by‑Step Process

1. User Requests Consent

Examples:

• “Can I record this for my journal?”
• “Is it okay if I capture this moment?”


2. Confirmation Received

Consent may be granted via:

• Verbal “yes”
• Nod or gesture
• Written event policy
• Parental approval


3. Device Creates ConsentEntry

• Generates content_hash
• Assigns context_tag
• Records consent_mode
• Signs entry with device key


4. Entry Stored Locally

Stored in encrypted local ledger.

5. Gatekeeper Validation

When upload is attempted, the Guardian Upload Gatekeeper checks:

• Does a matching ConsentEntry exist?
• Is it still valid?
• Does the context allow upload?
• Is the consent mode appropriate (e.g., parental)?


If yes → upload may proceed.
If no → upload blocked or restricted.

---

6. Consent Matching Logic

The Gatekeeper matches media to consent using:

\text{match}(H_{\text{media}}, H_{\text{ledger}}) = (H_{\text{media}} = H_{\text{ledger}})


Matching Requirements

• Exact hash match
• Context compatibility
• Consent mode compatibility
• Expiry not exceeded


Non‑Matching Conditions

• Different content hash
• Expired consent
• Wrong context
• Wrong consent mode
• Missing parental consent for minors


---

7. Context‑Aware Consent Rules

7.1 Sensitive Contexts

Consent cannot override restrictions in:

• Hospitals
• ICU
• Changing rooms
• Locker rooms
• Therapy offices
• Confessionals
• Private counseling spaces


Even with consent → upload permanently blocked.

---

7.2 Event‑Level Consent

For schools, sports, or public events:

• Event policy may grant group consent
• Individual consent still respected
• Child blur cannot be disabled without parental consent


---

7.3 Session‑Bound Consent

If expiry is null → consent applies only to the current session.

---

8. Security Requirements

8.1 Local Encryption

Ledger must be encrypted at rest.

8.2 Device‑Signed Entries

All entries must be signed using device private key.

8.3 Tamper Detection

Any modification invalidates signature.

8.4 No Cloud Sync

Consent entries never leave the device.

8.5 No Identity Leakage

Ledger must not be inferable as a list of people.
