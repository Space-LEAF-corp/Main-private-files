# Security Policy

The GQI&EA system is designed with transparency, traceability, and user safety in mind.  
This document outlines how to report vulnerabilities and how the project handles security concerns.

---

## Supported Versions

Security updates apply to:
- The main branch  
- Any actively maintained release branches  

Older or archived versions may not receive patches.

---

## Reporting a Vulnerability

If you discover a security issue, please report it responsibly.

### To report:
1. Do **not** open a public GitHub issue.  
2. Contact a project maintainer privately.  
3. Include:
   - A clear description of the vulnerability  
   - Steps to reproduce  
   - Potential impact  
   - Any suggested fixes (optional)

We will acknowledge receipt within a reasonable timeframe.

---

## Responsible Disclosure

We ask that you:
- Give maintainers time to investigate and patch the issue  
- Avoid sharing exploit details publicly  
- Avoid using the vulnerability for personal gain or harm  

We commit to:
- Reviewing reports promptly  
- Communicating respectfully  
- Issuing patches as needed  
- Crediting responsible reporters (if desired)

---

## Security Principles

The GQI&EA project follows these guiding principles:

- **Traceability:** All inquiries are timestamped and tagged.  
- **Minimal exposure:** No unnecessary data is stored or shared.  
- **Modularity:** Components are isolated to reduce risk.  
- **Transparency:** Security decisions are documented when appropriate.  
- **Non‑punitive reporting:** Good‑faith reports are always welcome.

---

## Scope

This policy applies to:
- Core logic  
- API modules  
- CLI and UI components  
- Documentation that affects security behavior  

It does **not** cover:
- Third‑party dependencies  
- User‑hosted deployments  
- External systems integrated by contributors  

---

## Thank You

Your efforts to keep this project secure help protect everyone who uses it.  
We appreciate your stewardship and responsible participation.
